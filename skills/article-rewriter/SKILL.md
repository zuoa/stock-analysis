# Article Rewriter

三模式文章改写技能：
- `quick`：只做翻译与重写，不主动外查
- `standard`：分析原文，必要时补充少量资料后重写
- `publish`：分析、核验、补充、重组、润色，产出可直接发布版本

默认模式：`standard`

## 输入类型

- `inline text`：用户直接粘贴正文
- `file path`：读取本地文件
- `URL`：抓取正文后再处理

## 默认值

| 设置 | 默认值 | 说明 |
|------|--------|------|
| 输出语言 | `zh-CN` | 最终统一输出中文 |
| 模式 | `standard` | 用户未指定时使用 |
| 平台 | `generic` | 若用户提到公众号/知乎/Newsletter，则改用对应预设 |
| 标题数 | `5` | 用户未指定时提供 5 个标题候选 |
| 调研范围 | `2-4` 个主题 | `publish` 模式可扩展到 `4-6` 个 |
| 输出目录 | `rewrites/{slug}-{timestamp}/` | 所有中间文件和最终稿统一放在这里 |

## 模式

| 模式 | 触发词 | 步骤 | 适用场景 |
|------|--------|------|----------|
| Quick | “简单润色”“快改一下”“不要查资料” | 翻译（如需要）→ 重写 | 短文、轻量润色、用户明确不要外查 |
| Standard | 默认 | 分析 → 选择性调研 → 重写 → 标题 | 大多数文章改写需求 |
| Publish | “深度改写”“公众号成稿”“适合发布”“补充资料并重写” | 分析 → 调研/核验 → 结构重组 → 重写 → 润色 → 标题 | 对成稿质量要求高、要发布或事实敏感的内容 |

自动判断规则：
- 用户明确说“不用查资料”“只改表达”时，用 `quick`
- 用户明确要“发公众号/知乎/Newsletter”“深度改写”“补资料”“像一篇正式文章”时，用 `publish`
- 其他情况默认 `standard`

## 输出目录

为每次任务创建输出目录：`rewrites/{slug}-{timestamp}/`

| 文件 | 模式 | 说明 |
|------|------|------|
| `source.md` | 全部 | 落盘后的原始内容 |
| `01-analysis.md` | `standard` / `publish` | 主题、受众、结构、必须保留的信息、薄弱点 |
| `01-terms.md` | 有翻译时 | 本次任务的术语表、推荐译法、禁用译法 |
| `02-research.md` | `standard` / `publish` | 外部资料、事实核验、冲突信息、可引用要点 |
| `03-outline.md` | `publish` | 新结构、大纲、标题角度、叙述顺序 |
| `rewrite.md` | 全部 | 最终改写稿 |

如果用户只要求聊天里直接返回结果，仍然默认保存 `rewrite.md`，并在回复里给出路径。

## 平台与文风

平台、段落节奏、标题风格不要写死在主文件里。按需阅读：
- [references/style-presets.md](references/style-presets.md)：公众号、知乎、Newsletter、通用版的文风和标题约束
- [references/glossary-en-zh.md](references/glossary-en-zh.md)：英文原文或中英混排材料中的特殊译法

选择逻辑：
- 用户明确指定平台时，优先使用对应预设
- 用户未指定平台时，先根据原文类型和目标受众判断；仍不明确则使用 `generic`
- 技术、商业、社会议题都可以有叙事感，但不要强行“鸡血化”或“爆款化”

## 术语处理

当任务包含以下任一情况时，必须先做术语统一，再进入改写：
- 原文是英文
- 原文是中英混排
- 原文涉及 AI、产品、商业、技术等高频专有名词
- 用户明确要求“翻译并改写”

处理顺序：
1. 优先读取内置术语表 [references/glossary-en-zh.md](references/glossary-en-zh.md)
2. 从原文额外提取专有名词、缩写、产品名、方法论名词
3. 形成本次任务的会话术语表
4. 如无必要，不在正文里反复切换同一术语译法

执行原则：
- 常见且约定俗成的词，直接用标准中文
- 容易误译、风格差异大的词，优先遵循内置术语表
- 首次出现时，可使用 `中文（English）` 或 `中文（英文原词，简短解释）`
- 如果术语存在多种合理译法，选择最适合目标平台和受众的一种，并全文保持一致
- 改写允许重组句子，但不要把关键术语“改丢”或改成过度口语化的说法

## 工作流

### Step 1: 获取约束并确定改写策略

优先从当前对话提取，不要为了默认值反复追问。确认以下信息：
- 输入来源：文本、文件、URL
- 用户目标：润色、扩写、翻译并改写、发文成稿、平台适配
- 是否允许外部检索
- 是否要保留原文结构
- 是否要控制篇幅
- 目标平台与受众

如果用户没有说明：
- 默认允许“必要时补充少量资料”，但不要为静态内容滥搜
- 默认允许重组结构，但要保留原文核心论点
- 默认提供 5 个标题候选

### Step 2: 落盘原文并创建输出目录

1. 将输入内容统一落盘到 `source.md`
2. 创建输出目录 `rewrites/{slug}-{timestamp}/`
3. 检测原文语言
4. 如果原文不是中文，先加载内置术语表并提取会话术语，写入 `01-terms.md`
5. 如果原文不是中文，先生成中文工作底稿，再继续后续步骤

要求：
- URL 输入优先抽取正文，避免导航、推荐位、评论区污染
- 文件或 URL 抽取失败时，明确说明阻塞点；如果仍可继续，退回到用户已提供的正文

### Step 3: 分析原文

`standard` 和 `publish` 模式需要产出 `01-analysis.md`，至少包含：
- 原文主题和核心论点
- 目标受众推断
- 原文结构拆解
- 必须保留的事实、数字、案例、引用
- 关键术语及建议译法（如果有翻译步骤，可引用 `01-terms.md`）
- 可优化点：过长、重复、跳跃、空泛、事实老旧、结论太早或太晚
- 改写策略：保守重写 / 结构重组 / 翻译后二次创作

如果原文已经写得很好，分析里要明确“以增量和可读性优化为主”，不要强行颠覆。

### Step 4: 调研与事实核验

何时需要做 `02-research.md`：
- 用户明确要求补充资料、最新信息、案例、数据
- 原文含有明显时效性内容
- 原文中的数字、研究、公司信息可能已经过期
- 主题本身需要最基本的事实校验

按需阅读：
- [references/research-rules.md](references/research-rules.md)：检索范围、来源优先级、冲突处理、引用格式

执行原则：
- `quick` 模式默认不主动外查
- 用户明确禁止检索时，跳过外查，并在最终结果里注明“本次基于原文改写，未补充外部资料”
- `standard` 模式只查能显著提高文章可信度的关键信息
- `publish` 模式要把“外部事实”“作者观点”“你的推断”分开

### Step 5: 规划新结构

`publish` 模式必须先写 `03-outline.md`，再落正文。至少包括：
- 标题方向：洞察型 / 利益型 / 反常识型 / 场景型 / 提问型
- 开头策略：故事、场景、反差、问题、结论先行
- 主体段落顺序
- 每段要完成的作用
- 结尾动作：总结、提醒、方法、下一步建议

`standard` 模式可以不单独落盘大纲，但心里必须先确定结构再写。

### Step 6: 重写正文

所有模式都遵守以下原则：

- **忠于原意**：保留核心观点、逻辑链和事实关系
- **增量优先**：新增信息必须提升理解、可信度或可读性，不能只是注水
- **中文自然**：像成熟作者写给人看，不像把原文逐句搬运
- **事实克制**：不要杜撰数据、引用、人物案例、调查结果、采访内容
- **推断显式**：推断就是推断，不要伪装成已核实事实
- **风格适配**：按平台、主题和受众调整叙述密度，别一律写成“公众号鸡汤”
- **段落节奏**：长短句交替，避免机械的“首先/其次/最后”
- **术语一致**：优先遵循 `01-terms.md`，技术词第一次出现时，必要时加简短解释

推荐结构：
1. 开头用一个问题、场景、结论或反差把读者拉进来
2. 迅速说明这篇文章要解决什么问题
3. 用 3-5 个核心段落展开论点
4. 在关键处插入数据、案例或反面证据
5. 结尾给出判断、边界或行动建议

### Step 7: 生成标题候选

默认提供 5 个标题，优先覆盖不同角度，而不是只换几个词：
- 洞察型
- 利益型
- 反常识型
- 场景型
- 提问型

标题规则：
- 必须准确反映正文核心价值
- 不伪造数字，不夸大结论，不制造不存在的紧迫感
- 平台越正式，标题越克制
- 同一组标题之间角度要明显不同

如果用户明确说“不需要标题”，则跳过。

### Step 8: 保存并回复

最终稿始终保存到 `rewrite.md`。

回复时给出：

```text
改写完成（{mode} 模式）

Source: {source-path}
Platform: {platform}
Research: {used / skipped}
Output dir: {output-dir}/
Final: {output-dir}/rewrite.md
```

## 最终输出格式

`rewrite.md` 默认使用以下结构：

```markdown
## 标题候选

1. 标题一
2. 标题二
3. 标题三
4. 标题四
5. 标题五

---

## 正文

[改写后的完整文章]

---

## 参考来源

- [来源标题](链接)
- [来源标题](链接)

---

## 改写说明

- 模式：standard
- 平台：wechat
- 是否补充外部资料：是
- 是否使用术语表：是
```

如果未外查，把“参考来源”改为：

```markdown
## 参考来源

- 本次基于用户提供原文改写，未补充外部资料
```

## 质量门槛

- 信息密度应高于原文，而不是仅仅换说法
- 如果扩写，新增内容要么解释关键概念，要么补足案例/数据/边界
- 事实敏感内容应优先核验，不确定时宁可保守表达
- 风格要有人味，但不要表演式抒情
- 如果原文质量本来很高，应以“精修”代替“重写”

## 禁止事项

- 不要把未经核实的信息包装成“最新研究表明”
- 不要为了制造传播感而扭曲结论
- 不要照搬大段原句，只做词语替换
- 不要堆砌空泛修辞、套话或过量感叹句
- 不要默认所有文章都适合“爆款标题”
